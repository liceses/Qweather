## 1. 简介

### 1.1 编写目的

本文档旨在对基于 Qt 6.11 框架 + QML 图形界面开发的桌面天气预报应用 **QWeather** 进行全面、系统的需求规格说明，明确项目的开发目标、功能范围、模块划分和设计约束，为后续维护、迭代和协作提供统一的参考基线。

本应用通过调用和风天气（QWeather）公开 API，提供多城市天气信息的便捷查询与可视化展示服务，并集成 GPU 驱动的动态天气背景系统，为用户提供沉浸式的天气信息浏览体验。

### 1.2 使用范围

本项目面向日常用户、学生、旅游爱好者以及需要获取专业天气信息的各类人群，可运行于 **Windows** / **Linux** 桌面平台。适用场景包括：

- 个人日常天气查询与多城市对比
- 教学演示与技术展示（Qt Quick / QML / ShaderEffect 实战）
- 天气数据可视化与天文信息展示
- 动态图形渲染（GPU 粒子系统、实时天空模拟）的技术验证

**软件核心能力**：

- 基于 Qt 网络模块调用和风天气 API，实现城市搜索、实时天气、7 天/30 天预报、空气质量指数、太阳辐射与天文数据查询
- 前端采用 QML + Qt Quick 设计响应式 Glassmorphism 界面，全部天气图标通过 CDN SVG 动态加载
- 数据缓存使用 SQLite，减少重复网络请求
- 集成 GPU ShaderEffect 实现动态天气背景（晴/阴/雨/雪/雾/夜 + 太阳/月亮天文天空模型）
- 集成 QtCharts 绘制温度趋势折线图，支持多预报范围切换

## 2. 总体概述

### 2.1 软件概述

**QWeather** 是一款基于 **Qt 6.11** 框架开发的桌面天气预报应用，采用 **C++ 后端 + QML 前端** 的混合架构。

- **后端（C++）**：统一天气 API 引擎（24 个 API 端点，统一请求/路由/缓存/信号体系）、4 个数据 Store（Forecast/AirQuality/SolarAstronomy/CityDetail）、全局时钟、背景管理器与过渡控制器
- **前端（QML）**：7 页面路由（仪表盘/预报/空气质量/阳光天文/城市详情/收藏/设置）+ 6 层动态天气背景 Layer 栈（ShaderEffect GPU 渲染）+ 响应式 UI 组件
- **数据流模式**：C++ Store 预处理 JSON → 通过 `contextProperty` 注入 QML 引擎 → QML 纯绑定渲染（QML 端不解析 JSON）

## 3. 功能需求

### 3.1 用户功能

#### 3.1.1 城市搜索与定位模块

- 用户在搜索框输入城市名称 → 系统触发 `cityLookup` API 返回联想结果列表（Popup 展示）
- 用户点击联想项或按下回车 → 发起指定城市的天气查询
- 支持通过 `GeoAPI` 获取城市经纬度、LocationID、时区等信息
- 用户首次启动时，自动加载热门城市列表填充仪表盘

#### 3.1.2 仪表盘主界面

- 顶部胶囊形搜索框
- 焦点城市大字 + 实时天气摘要（温度/体感温度/天气描述/高低温范围）
- 信息芯片行：体感温度、湿度、风力风向、紫外线指数、能见度、气压
- 城市卡片行（水平排列，≤4 张）：城市名 + 天气图标 + 实时温度
  - 单击 → 切换焦点城市
  - 双击 → 跳转城市详情页

#### 3.1.3 导航与页面切换

- 左侧边栏导航按钮（7 个页面入口）：仪表盘 / 天气预报 / 空气质量 / 阳光天文 / 设置 / 城市详情 / 收藏
- 侧边栏支持 hover 展开（80px ↔ 200px），200ms OutCubic 动画
- 页面切换通过 StackLayout 管理，NavButton 高亮同步

#### 3.1.4 收藏与固定

- 城市详情页星号图标 → 点击收藏城市
- 收藏页面：列表展示所有收藏城市，支持固定到侧边栏
- 双击收藏卡片 → 跳转城市详情
- 收藏/固定数据通过 SQLite 持久化保存

### 3.2 系统功能

#### 3.2.1 天气数据引擎模块

- 封装 **24 个和风天气 API 端点**，涵盖地理信息、实时天气、逐日/逐时预报、格点预报、空气质量、太阳辐射、天文数据、天气指数、天气预警、台风、潮汐、历史时光机
- 统一请求入口 `sendRequest()` + 统一回复路由 `onReplyFinished()` + 按 `_loc` 追踪参数分派
- 缓存拦截：`sendRequest()` 前检查 `WeatherCache`，命中且未过期则直接返回缓存 JSON
- API Key 认证：通过 URL 查询参数 `key` 传递（如 `?key=xxxxxxxx`），当前采用和风天气开发者 API Key 模式（待迁移至 JWT Bearer Header 模式）
- 全部信号签名统一为 `(QString cityId, QJsonObject data)` / `(QString cityId, QJsonArray data)`

#### 3.2.2 C++ 数据预处理层

- 各 Store 组件负责：请求触发 + JSON 解析 + 数据转换 + 触发 QML 绑定更新
- 核心模式：`setCities()` → `refreshAll()` → 逐个调用 API → Handler 解析 → 写 QVariantMap → `emit dataChanged()`
- 关键数据流：
  - 天气预报：JSON → 图表坐标数组 `[{x: idx, y: temp}]`
  - 空气质量：解析 `indexes[].aqi`、`pollutants[]`、颜色转换
  - 阳光天文：3 路异步 API（solar/sun/moon）→ 中间累加 → 合并写入
  - 城市详情：10 路 API 汇聚为单一 QVariantMap

#### 3.2.3 SQLite 缓存管理模块

- 键值存储（Key-Value），序列化 JSON 字符串
- TTL（生存时间）过期策略，不同类型数据设定不同有效期
- `save(key, value)` / `load(key)` 持久化接口
- 应用级别：收藏城市、固定城市、历史记录的 JSON 持久化

#### 3.2.4 动态背景渲染引擎（V3 天空模拟系统）

- **Layer 栈架构**：6 个 ShaderEffect 层按条件叠加渲染
  - SkyLayer（z=0）：天空渐变 + 太阳光晕 + 月亮 SDF + 星空粒子
  - AtmosphereLayer（z=10）：twilight 色调、散射、曝光
  - CloudLayer（z=20）：2~3 层 fBM 云噪声（条件：cloudCoverage > 0.01）
  - WeatherLayer（z=30）：雨丝/雪花粒子 ≤60（条件：precipIntensity > 0.001）
  - FogLayer（z=40）：雾/霾/沙尘色调（条件：fogDensity > 0.001）
  - LightningLayer（z=50）：间歇全屏闪电闪烁（条件：lightningProb > 0）
- **三层过渡模型**：
  - 层 0（NO-OP）：完全相同 → 无操作
  - 层 1（参数平滑）：同类型 intensity/variant 变化 → 400ms NumberAnimation
  - 层 2（过程式过渡）：跨类型切换 → 双层 Loader + 800ms transitionProgress 驱动元素级动画
- **防抖机制**：过渡期间新请求 → abortCurrentTransition() → 恢复状态 → 重发

#### 3.2.5 天文天空模型

- 基于日出日落时间计算 `sunProgress` 驱动 7 时段天空色插值
- 太阳位置：`angle = (sunProgress - 0.5) × π` 计算 sunX/sunY/sunGlow
- 月亮位置：与太阳对跖（相位差半圈），可见性仅限夜晚+twilight
- 月相：800（新月）~807（残月），SDF 裁剪月牙形状
- 天文数据 60s 定时器刷新，计算开销 <0.1ms

#### 3.2.6 性能分级策略

| 档位 | Rain 粒子 | Snow 粒子 | Night 星星 | 适用场景 |
|:---:|:---------:|:---------:|:---------:|---------|
| 高(2) | 60 | 60 | 50 | 桌面独显 |
| 中(1) | 40 | 40 | 35 | 集显/笔记本 |
| 低(0) | 20 | 20 | 20 | 嵌入式 |

- 循环上限编译时常量（60），uniform break 控制实际数量
- 过渡期间 tp=0 层自动跳过粒子循环，不增加 GPU 负载

## 4. 功能模块设计

本天气预报应用围绕 **"实用性、交互性、美观性"** 三大核心原则，设计并实现了 10 个功能模块，涵盖天气信息查询、数据可视化、动态图形渲染、用户系统等全方位功能。

### 4.1 城市天气查询模块

**对应功能**: F1

#### 架构

```
用户输入 → SearchBar 300ms 防抖 → weatherApi.cityLookup()
  → 解析 location[] → Popup 联想列表
  → 用户选择 → root.addFocus(id, name, lat, lon)
    → weatherApi.weatherNow(focusId) → 实时天气
    → weatherApi.weatherDaily(3d, focusId) → 今日/明日预报
    → 各 Store 同步更新
```

#### 核心组件

| 文件 | 角色 | 说明 |
|------|------|------|
| `SearchBar.qml` | 前端组件 | 胶囊形输入框 + Popup 联想面板，300ms 防抖 + 键盘上下选择 |
| `weatherApi.h/.cpp` | 后端 | `cityLookup()` 地理编码 API + `weatherNow()` 实时天气 API |
| `Main.qml` | 全局状态 | `focusId / weathers[]` 管理焦点城市与所有城市数据 |

#### 关键行为

1. **首次启动**：无历史城市 → `weatherApi.topCity("cn", 4)` → 热门城市填充仪表盘
2. **焦点城市变化**：`onFocusIdChanged` → 实时天气刷新 + 各 Store 同步 + 历史写入
3. **搜索回车**：取消前一次请求，确保搜索当前输入内容

#### 数据流

```
SearchBar.selectCity(id, name, lat, lon)
  → root.addFocus(id, name, lat, lon)
    → onFocusIdChanged
      → weatherApi.weatherNow(focusId)       [实时天气]
      → cityDetailStore.setCity(id, name, lat, lon)  [城市详情同步]
      → addHistoryEntry(focusId)              [历史记录写入]
```

### 4.2 七天天气预报模块

**对应功能**: F2

#### 架构

```
Main.qml cityList 变更
  → forecastStore.cities = root.cityList
    → Store::setCities() → refreshAll()
      → 逐个 weatherDaily(range, cityId) / weatherHourly(range, cityId)
        → weatherDailyReady / weatherHourlyReady 信号
          → ForecastStore handler 解析 JSON
            → 提取 daily[].tempMax/Min 或 hourly[].temp
            → 生成坐标数组 [{x: 序号索引, y: 温度值}]
            → 写入 chartData + info → emit chartDataChanged()
              → ForecastCard.points 绑定重求值
                → onPointsChanged → updateChart() 渲染
```

#### 核心组件

| 文件 | 角色 | 说明 |
|------|------|------|
| `ForecastStore.h/.cpp` | C++ 数据层 | cities/range/mode/chartData Q_PROPERTY，JSON→坐标预处理 |
| `ForecastPage.qml` | 页面 | 8 选项范围选择器 + 2×2 Grid 布局，纯 UI |
| `ForecastCard.qml` | 卡片 | ChartView（预声明 SplineSeries + 2×LineSeries）+ 城市信息 |

#### 范围选择器

- **逐小时**：24h、72h、168h → `weatherApi.weatherHourly()`
- **逐日**：3d、7d、10d、15d、30d → `weatherApi.weatherDaily()`
- 后缀 `h`/`d` 判定模式，切换时 `ForecastStore.setRange()` → `refreshAll()`

#### 图表渲染方案

- ChartView 预声明 SplineSeries（温度曲线）+ LineSeries×2（最高温/最低温包络线）
- X 轴用 ValueAxis + 序号索引 (0, 1, 2...)，Y 轴自适应范围
- `updateChart()`：计算 yMin/yMax + 10% 边距 → `setRange()` → `clear()` → `append()`

### 4.3 空气质量指数模块

**对应功能**: F3

#### 架构

```
Main.qml cityList (含 lat/lon)
  → airQualityStore.cities = root.cityList
    → refreshAll()
      → 逐个 airCurrent(lat, lng, cityId)
        → airCurrentReady(cityId, JSON) 信号
          → AirQualityStore.onAirCurrentReady()
            → 解析 indexes[us-epa].aqi / level / category / color
            → 解析 primaryPollutant
            → 解析 pollutants[].concentration
            → 写入 airData[cityId] = {aqi, category, color, pollutants...}
            → emit airDataChanged()
              → AirQualityCard 绑定刷新
```

#### 核心组件

| 文件 | 角色 | 说明 |
|------|------|------|
| `AirQualityStore.h/.cpp` | C++ 数据层 | cities/airData Q_PROPERTY，解析 AQI JSON → QVariantMap |
| `AirQualityPage.qml` | 页面 | 2×2 Grid 布局，绑定 airQualityStore |
| `AirQualityCard.qml` | 卡片 | AQI 大数字 + 彩色圆点 + 类别标签 + 首要污染物 + 污染物浓度网格 |

#### 显示指标

- AQI 数值与等级（US EPA 标准）
- 空气质量类别（Good/Moderate/Unhealthy 等）与颜色指示
- 首要污染物标识
- 污染物网格：PM2.5（μg/m³）、PM10（μg/m³）、NO₂（ppb）、O₃（ppb）、CO（ppm）

### 4.4 阳光天文信息模块

**对应功能**: F4

#### 架构

```
Main.qml cityList
  → solarAstronomyStore.cities = root.cityList
    → refreshAll()
      → 每城 3 路 API 并行:
        solarRadiation(lat, lng, cityId)    → forecasts[0] → ghi/dni/dhi/solarAngle
        astronomySun(locId, date, cityId)    → sunrise/sunset
        astronomyMoon(locId, date, cityId)   → moonPhase[0] → name/icon/illumination
    → m_solarRaw[cityId] 中间累加器（3 路数据到达标记）
      → 全部到达 → mergeAndEmit() 合并写入 m_solarData[cityId]
        → emit solarDataChanged()
          → SolarAstronomyCard 绑定刷新
```

#### 核心组件

| 文件 | 角色 | 说明 |
|------|------|------|
| `SolarAstronomyStore.h/.cpp` | C++ 数据层 | 3 路异步 API 合并，中间累加 → 统一 emit |
| `SolarAstronomyPage.qml` | 页面 | 2×2 Grid，标题随 `appSettings.showSolarRadiation` 切换 |
| `SolarAstronomyCard.qml` | 卡片 | GHI 大数字 + DNI/DHI + 日出/日落 + 月相名称/亮度 |

#### 关键设计

- 3 路 API 返回时序不确定 → `m_solarRaw` 中间缓冲 + 位标记追踪到达状态
- 全部到达后 `mergeAndEmit()` 一次性写入，避免 QML 侧多次刷新
- 月相图标码（800-807）用于月牙 SDF 裁剪

### 4.5 城市详情聚合模块

**对应功能**: F5

#### 架构

```
Main.qml focusId 变更
  → cityDetailStore.setCity(id, name, lat, lon)
    → 并行调用 10 路 API:
      weatherNow / weatherDaily(3d) / weatherHourly(24h)
      airCurrent / indices(1d,1,2,3,5,6,8,9) / warningNow
      astronomySun / astronomyMoon
      solarRadiation / minutelyPrecip(5m)
    → 各 Handler 解析 JSON 字段 → 写入 m_detail QVariantMap
    → emit detailChanged()
      → CityDetailPage 9 张卡片按数据到达依次可见
```

#### 核心组件

| 文件 | 角色 | 说明 |
|------|------|------|
| `CityDetailStore.h/.cpp` | C++ 数据层 | 10 路 API 汇聚为单一 QVariantMap，构造函数中预初始化所有嵌套 map/list |
| `CityDetailPage.qml` | 页面 | Flickable 滚动 + 9 张卡片堆叠（实时/空气/逐日/逐时/指数/预警/天文/辐射/降水） |

| 空气质量 | AQI、等级、污染物、健康建议 | `airCurrent` |

**对应功能**: F6

### 4.7 图表渲染与数据可视化模块

**对应功能**: F10

#### 架构

- **工具**：Qt 6 Charts（`QtCharts` 模块）
- **部署**：`ForecastCard.qml` 内嵌 `ChartView` 组件
- **Series**：
  - `SplineSeries`：平滑温度曲线
  - `LineSeries` ×2：最高温包络线 / 最低温包络线
- **Axis**：`ValueAxis` + 序号索引（避免 DateTimeAxis 兼容性问题）
- **渲染流程**：
  ```
  C++ 端: JSON → [{x: idx, y: temp}] → emit chartDataChanged()
  QML 端: onPointsChanged → 计算 yMin/yMax + 10% padding
        → valueAxis.setRange(yMin, yMax)
        → series.clear() → series.append(points)
  ```

#### 关键约束

- 依赖 `QApplication`（非 `QGuiApplication`），因 QtCharts 内部使用 QGraphicsTextItem→QWidgetTextControl
- 预声明 series 优于动态 `createSeries()`，避免轴进入无效状态

### 4.9 用户系统（收藏/历史/焦点城市）

**对应功能**: F7

#### 焦点城市行为

```
首次启动 → 无历史 → topCity("cn", 4) → cityList ← 热门城市
  → focusId = cityList[0].id
  → onFocusIdChanged → weatherNow + cityDetailStore.setCity

单击 CityCard → root.addFocus(cityId)
  → 已在列表 → 仅切焦点
  → 不在列表 → 替换焦点城市卡片
双击 CityCard → focusId + stack.currentIndex = 5 (CityDetailPage)
SearchBar 选择 → 新城市替换当前焦点城市卡片
```

#### 侧边栏城市优先级

```
pinned[0].id > favorites[0].id > history[0].id > focusId
```

#### 收藏与固定

- **收藏**：城市详情页星号图标点击 → 加入 `root.favorites` 数组 → SQLite 持久化
- **固定**：收藏页面每个卡片有固定按钮 → 加入 `root.pinned` 数组 → 侧边栏城市分区展示
- **历史**：每次焦点城市变化自动记录 → LIFO 去重 + 上限 20 条 → SQLite 持久化

#### 持久化路径

```
收藏/固定/历史 → JSON.stringify
  → weatherCache.save("fav_cities"/"pin_cities"/"hist_cities", json)
启动时 → weatherCache.load(...) → JSON.parse → 恢复到对应数组
```

## 5. 总体设计约束

### 5.1 技术限制

| 类别 | 约束 |
|------|------|
| **开发框架** | Qt 6.11+（最低要求 6.10，使用 CMake 构建） |
| **编程语言** | C++17（后端）、QML/JavaScript（前端）、GLSL（着色器） |
| **构建工具** | CMake 3.16+，MinGW 64-bit 工具链 |
| **图形 API** | Qt RHI 抽象层（Direct3D 11 / OpenGL 3.3+ / Vulkan 1.1+） |
| **Chart 引擎** | Qt 6 Charts 模块（依赖 QtWidgets） |
| **网络通信** | Qt Network Access Manager，HTTPS |
| **数据库** | SQLite 3（通过 Qt SQL 模块） |
| **目标平台** | Windows 10/11 64-bit、Linux (X11/Wayland) |
| **API 服务** | 和风天气 API（devapi.qweather.com / api.qweather.com） |
| **API 认证** | API Key 查询参数模式（`?key=xxxx`，待迁移至 JWT Bearer Header 模式） |

### 5.2 架构约束

1. **C++ → QML 单向数据流**
   - C++ Store 是唯一数据源，QML 端只做绑定渲染，不解析 JSON
   - 所有 Store 通过 `QQmlApplicationEngine::rootContext()->setContextProperty()` 注册

2. **QML 属性绑定规则**
   - C++ 端每次创建新 `QVariantMap` 对象以触发 QML Changed 信号
   - 所有 `visible` guard 表达式必须使用 `!!` 强制返回布尔值（`undefined && x` 返回 `undefined` 非 `false`）
   - C++ 端预初始化所有嵌套 map/list 为空对象，防止 QML 绑定提前求值导致 TypeError

3. **QtCharts 约束**
   - 必须使用 `QApplication` 而非 `QGuiApplication`
   - 预声明 series 优于动态 `createSeries()`
   - `clear()` 前必须先 `setRange()` 设置轴范围

4. **Shader 部署**
   - GLSL 源文件通过 `qt_add_shaders()` 自动编译为 `.qsb`（Qt Shader Baker）格式
   - 所有着色器内置 fallback Rectangle 渐变，在 ShaderEffect Error 时显示

5. **模块间通信**
   - Store 与 Store 之间不直接调用，全部通过 Main.qml 的事件转发
   - API 请求通过 `_loc` 查询参数追踪请求所属城市，避免异步返回时路由丢失

6. **性能约束**
   - GPU 粒子循环上限编译时常量（60），通过 uniform 变量控制实际数量
   - 天文计算每 60s 一次，单次 <0.1ms
   - 过渡期间跳过粒子循环，不翻倍 GPU 负载
